# HiveLens

Fan segmentation & AI campaign studio. Ingest a fan list, auto-build audience
segments (VIP, lapsed, first-timer, high-spender, local), build custom segments
with live filters, and generate personalized email + SMS campaigns per segment.

- **Backend:** Python 3.11+ / Flask, PostgreSQL for demo data
- **Frontend:** React 18 + TypeScript
- **AI copy:** Claude (`claude-sonnet-4`) when an API key is present, otherwise
  a library of pre-generated campaigns served from Postgres

---

## Two ways to run it

HiveLens is designed so you can demo the full experience **before paying for an
API key**, then flip to live Claude generation later with zero code changes.

| Mode | Requires | Campaign copy comes from |
|------|----------|--------------------------|
| **Demo mode** (default) | A local Postgres + seed | Pre-generated copy stored in the `campaigns` table. Each "Regenerate" rotates through multiple hand-tuned variants per segment, so it feels live. |
| **Live mode** | `ANTHROPIC_API_KEY` | Real-time Claude generation. The DB is still used for the 2,000-fan dataset. |

The backend picks the mode automatically: if `ANTHROPIC_API_KEY` is set it calls
Claude; otherwise it serves the pre-generated library from the database. The
campaign preview drawer labels the source ("Generated by Claude" /
"Pre-generated copy from demo database").

> If Postgres itself is unreachable, the backend transparently falls back to a
> 500-fan in-memory fixture so the app still boots — but the **database demo
> path below is the recommended easy setup.**

## Prerequisites

- Python 3.11+
- Node 18+
- PostgreSQL 14+ running locally (the app connects to `localhost:5432`)

## Quick start

### 1. Backend + database (demo data)

```bash
cd hivelens/backend

# create a virtualenv and install deps
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt

# create the database (one time)
createdb hivelens

# seed it: 2,000 fans + the pre-generated campaign library
.venv/bin/python seed.py
# -> Seed complete:
#      fans:      2000
#      campaigns: 102 (6 events x 6 segments)

# run the API on http://localhost:5000
.venv/bin/python app.py
```

Want more data? `FAN_COUNT=5000 .venv/bin/python seed.py`.

### 2. Frontend

```bash
cd hivelens/frontend
npm install --legacy-peer-deps      # see Troubleshooting for why
npm start                           # http://localhost:3000
```

### 3. Use it

1. Open http://localhost:3000
2. Pick a **demo event** from the dropdown (Rolling Loud LA, Electric Forest
   NYC, Lollapalooza Chicago, Miami Bass Festival, Austin City Limits, Toronto
   Jazz Nights) and click **Load Demo Event** — 2,000 fans load from Postgres.
3. Five auto-segments populate with live counts.
4. Click **Generate Campaign** on any segment → the drawer shows a personalized
   email + SMS pulled from the database. Hit **Regenerate** to cycle variants.
5. Use the **SegmentBuilder** to filter (e.g. EDM fans in LA who spent >$200),
   save it as a custom segment, and generate a campaign for it too.

## Switching to live Claude generation

When you're ready to pay for the API key, no code changes are needed:

```bash
export ANTHROPIC_API_KEY=sk-ant-...
# restart the backend
.venv/bin/python app.py
```

The backend now routes campaign generation to Claude (`claude-sonnet-4`). The
2,000-fan dataset still comes from Postgres.

## Configuration

| Env var | Default | Purpose |
|---------|---------|---------|
| `DATABASE_URL` | `dbname=hivelens` (local peer auth) | Postgres connection string. Example: `postgresql://user:pass@localhost:5432/hivelens` |
| `FAN_COUNT` | `2000` | Number of fans generated by `seed.py` |
| `ANTHROPIC_API_KEY` | _unset_ | When set, enables live Claude generation |

---

## Data model (Postgres)

- **`fans`** — the synthetic fan dataset. Money is stored as integer cents
  (`total_spend_cents`, `avg_ticket_spend_cents`); `genres` is a `text[]`.
- **`campaigns`** — pre-rendered email/SMS copy keyed by
  `(segment, event_name, variant_idx)`, with `projected_open_rate`. This is the
  "AI-feel" content library that powers demo mode.

Re-running `seed.py` truncates and rebuilds both tables.

## API reference

| Method & path | Body | Returns |
|---------------|------|---------|
| `POST /api/ingest` | `{"use_fixture": true}` or `{"fans": [...]}` | `{fan_count, source}` |
| `POST /api/segments` | `{event_lat, event_lng}` | auto-segment counts + fan ids |
| `POST /api/segment/custom` | `{filters: {...}}` | `{count, fan_ids, sample}` |
| `GET  /api/fans/sample` | `?segment=vip&limit=5` | sample fans |
| `POST /api/campaign/generate` | `{segment, event_name, event_date, event_city, event_lat, event_lng}` | `{subject_line, email_body, sms_body, send_estimate, projected_open_rate, source}` |
| `GET  /api/health` | — | `{db_available, db_fan_count, fan_source, ...}` |

## Troubleshooting

- **`npm install` peer-dependency error** — `react-scripts@5` declares a
  TypeScript ^4 peer while we use TS 5. Install with `--legacy-peer-deps`.
- **`Cannot find module 'ajv/dist/compile/codegen'`** during `npm run build` —
  a known react-scripts hoisting issue. Fixed by the pinned `ajv@8` in
  `package.json`; if it recurs, run `npm install ajv@8 --legacy-peer-deps`.
- **Backend `Client.__init__() got an unexpected keyword argument 'proxies'`** —
  `anthropic==0.30.0` is incompatible with newer `httpx`. `requirements.txt`
  pins `httpx<0.28` to fix it.
- **`could not connect to server: ... 5432`** — Postgres isn't running or
  `DATABASE_URL` is wrong. The app will fall back to the in-memory fixture, but
  for the full demo start Postgres and re-run `seed.py`.
